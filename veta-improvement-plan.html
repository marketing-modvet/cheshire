1. 🐛 The Hallucination Problem

From simulation results and what we've seen in testing, hallucinations cluster into five repeatable patterns. None of them are random — each has a fixable cause.

Pattern A — Confident wrong-brand routing

Example: User asks about a rabbit; bot confidently lists Modern Vet branches and "exotic specialist Dr. Ionut" without the BOV/VV alternative or the surgery-vs-consultation distinction.

Cause: Brand-separation rules live in SOUL.md (prompt) but aren't enforced at retrieval. The retriever can pull a single MV-only entry and the LLM speaks confidently from it.

Fix: Brand-aware retrieval (filter chunks by detected species/intent before the LLM sees them) + a hard post-check.

Pattern B — Branch-blind pricing

Example: "How much is a consultation?" → "AED 199-299". Technically true but useless. Then user books at JLT expecting AED 199 (Hospital is AED 299).

Cause: Bot answers without knowing which branch. Pricing is per-branch but the slot extractor doesn't pin one down before pricing answers.

Fix: Mandatory slot-filling — bot must ask "which branch?" before quoting price, or default to a single branch and SAY so.

Pattern C — Distance/geography invention

Example: "JVC is 3-4 km from Falcon City" (real distance: 15-18 km).

Cause: 127 neighbourhood mappings exist, but if the user's area isn't on the list, the LLM estimates from training data.

Fix: Hard rule — if neighbourhood not in neighbourhood_kb_entries.json, never quote distance. Either ask for area name, or list nearest 3 branches without distances.

Pattern D — Service-availability assumption

Example: Recommending Mobile Clinic for a surgery (Mobile only does consultations).

Cause: Branch-service matrix is not enforced; bot reasons "Modern Vet does surgery, Mobile is Modern Vet, ergo Mobile does surgery."

Fix: Explicit branch × service capability table, retrieved alongside any service-related answer.

Pattern E — Free-form medical advice

Example: User describes symptoms; bot says "could be X or Y, try Z" instead of routing to consultation.

Cause: When KB has no match, the model falls back on training data instead of escalating.

Fix: Confidence threshold — below 0.55 cosine similarity, bot must say "let me connect you with our team" and never offer medical specifics.

Common root cause

Veta's KB is rich, but the guardrails are in the prompt instead of the pipeline. A long instruction in SOUL.md saying "never make things up" loses to a confident-sounding retrieved chunk + a helpful LLM. We need to move guardrails from prompt-level to code-level.

Detail on the "anti-hallucination trio" (#1, #3, #5)

#1 — Confidence threshold + auto-escalation

Today, every retrieved chunk reaches the LLM regardless of similarity. The bot then writes a confident answer from a 0.7-distance match (i.e. weak).

New behaviour:

  • Top-1 distance < 0.40 → answer with full confidence
  • Distance 0.40–0.55 → answer + soft hand-off offer ("would you like me to connect you with our team to confirm?")
  • Distance > 0.55 → no answer attempt → "I'm not 100% sure about that — let me connect you with our team!" + auto-log as GAP

Effort: 30 lines of code in query_kb.py. Test: re-run 100 questions and see how often it correctly escalates.

#3 — Answer-grounding verifier

Run the bot's draft answer through a second Haiku call: "Here's the answer. Here are the retrieved chunks. Does every factual claim in the answer (price, location, doctor name, hour, distance) appear in the chunks? Y/N." If N → strip the unsupported claim or escalate.

Cost: ~$0.0003 per message. Cuts hallucinations 60-80% in published RAG benchmarks.

#5 — Intent + slot-filling pre-step

Before retrieval, run a tiny classifier that fills in:

  • Intent: booking / pricing / emergency / location / general / medical / complaint
  • Branch: JLT / Hospital / JVC / Palm / Downtown / etc / unknown
  • Species: dog / cat / rabbit / bird / exotic / unknown
  • Brand match: MV / BOV / VV / unspecified

Output drives filtered retrieval — instead of searching all 974 docs, search only the relevant slice. Massively cuts wrong-brand and wrong-branch errors.

4. 🔄 Better Workflow: How We Feed Veta

Right now the workflow is informal — Andrew finds a gap, tells Cheshire, Cheshire updates the KB, sometimes nothing reaches Aida or AR. This breaks down at scale. Proposed workflow:

📥 Single intake form / sheet

One Google Sheet (veta-kb-updates) where any of the three of you can drop:

  • New Q&A pair
  • Correction to existing answer
  • Reported hallucination from a real user
  • New gap discovered

Columns: Date · Submitter · Type · Q · A · Source · Approved by · Status

👥 Roles & approval flow

  • Andrew — submits anything client-facing, marketing, brand
  • Aida — submits + approves medical/clinical/operational
  • AR — submits + approves anything tech/integration/data
  • Cheshire — converts approved rows to KB entries, runs sims, reports back

Rule: medical answers require Aida's tick before going into KB. No exceptions.

🔁 Weekly review cycle

Every Monday Cheshire posts a Slack thread with:

  • New Q&A added that week
  • Hallucinations caught by the verifier
  • Open gaps still missing input
  • Confidence-score histogram (are we improving?)

10 minutes for the three of you to scan, comment, approve.

🚨 Live-traffic feedback loop

Once live, every conversation is logged. Each week, Cheshire pulls a random 30-message sample. Aida spot-checks for medical accuracy, Andrew for tone/brand, AR for technical issues. New gaps from real users flow back into the intake form.

Why this is "better than now": Today the chain has one bottleneck (Andrew → Cheshire). With three people contributing in their own area + a structured queue, gaps close 3-4× faster, medical accuracy is enforced, and AR sees what's coming so integration work doesn't get sprung last-minute.
