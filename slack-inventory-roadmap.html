1. Summary

Sinergi's spec proposes a structured Slack application (not a chatbot) that lets branch staff submit inventory requests through searchable product selectors, routes those requests to the correct category manager, and tracks the lifecycle from submission to PO/TO fulfilment in EzyVet. The audit trail lives in a Google Sheet repository (the 31-column template Sinergi provided).

Modern Vet's proposal: we build the application over 6 weeks starting the week of 8 June 2026, deploy it to dedicated independent infrastructure that we help set up, and hand it over to Sinergi + Michael's team for ongoing operation at go-live. EzyVet stays the system of record throughout, exactly as the spec specifies.

Target outcome: WhatsApp inventory groups sunsetted by mid-July. Branch staff submit through Slack. Pauline, Ivan, Aneesh approve through Slack. Procurement creates POs/TOs in EzyVet against approved request lines. Full audit trail in Google Sheets.

4. Why these tools

Slack Bolt over alternatives

Bolt is Slack's official framework and handles the parts of this build that matter most: the modal-based request form, the three interactive decision buttons on approver notifications, and the block-kit rich messages for status updates. Building any of these on the raw Slack Web API would mean re-implementing a lot of plumbing Bolt provides for free.

Google Sheets for the repository

Sinergi's spec calls for it, and we agree: it's the right choice for the MVP. Ops staff can read the audit trail without dev help, the user-location mapping can be edited without redeploying, and the 31-column schema works comfortably up to ~10k rows. Migration path to PostgreSQL later is a straightforward dump-and-import if volume ever demands it.

Manual EzyVet handoff, not API integration, for MVP

This is the single biggest scope-reducer in our proposal. Procurement continues to create POs and TOs in EzyVet exactly the way they do today — the app just records the EzyVet reference number back into the Slack flow via a "Mark as PO/TO Created" button on approved request lines. If EzyVet exposes a documented API we'd evaluate automating this in Phase 4, but we don't need it for the MVP to be useful.

Fuse.js for product search

Slack's native select menus struggle once you have more than a few hundred items. Branch staff need to find "Amoxicillin 500mg" by typing "amox" — Fuse.js handles that locally with sub-50ms latency, rebuilt from the product master every 5 minutes.

Dedicated independent hosting

We'll help set this up, but it's deliberately separate from any other Modern Vet workload. That gives Sinergi clean ownership of credentials, billing, and operational control from day one. Render and Railway are the easiest to operate (managed PaaS); a small Hetzner VPS is the cheapest if Sinergi prefers full control.

6. Build plan — 6 phases over 6 weeks

All dates assume Phase 0 kick-off on Monday 8 June 2026, with the week of 1 June used for the alignment call and final pre-work. The plan was compressed from an earlier 9–10 week draft by merging the form + approval phases, recording training material once and reusing it per branch, and fusing pilot with full rollout in the final week.

Phase 0 — Pre-build setup Jun 08–12 (5 days)
  • Michael exports product master from EzyVet → Google Sheet
  • Michael drafts user + branch mapping; we add Slack IDs
  • We help Sinergi/Michael provision dedicated hosting (Render, Railway, or Hetzner VPS)
  • Slack app created in Vet Alliance workspace with the required scopes
  • GitHub repo set up, hello-world Slack app deployed to confirm everything is wired
  • Sinergi + Modern Vet kick-off call

Exit criteria: all pre-requisites ready, hello-world app posts a test message to a Slack channel.

Phase 1 — Core MVP Jun 15–26 (2 weeks)

Week 1 — Foundations + request form

  • Slack app scaffolding (Bolt)
  • Google Sheets connector: read product master + user mapping into memory; write to request repository
  • Fuse.js product search index, refreshed every 5 min
  • /inventory slash command opens the request modal
  • Modal: product search → quantity → urgency → branch → notes
  • On submit: row written to Google Sheet, Request ID generated, status = Submitted
  • SKU / Category / Subcategory / UOM / Assigned Approver auto-filled from product master
  • Confirmation DM to requester

Week 2 — Approval flow

  • Approver receives interactive notification with Reject / Transfer / Purchase buttons
  • Each decision opens its own modal for required fields (rejection reason, transfer source & quantity, purchase quantity & ETA)
  • "Prepared By" auto-filled from the approver's Slack identity
  • Status transitions: Submitted → Approved/Rejected with requester notification

Exit criteria: end-to-end happy path works for one product, one approver, one branch.

Phase 2 — Operational queues & status tracking Jun 29 – Jul 10 (2 weeks)

Week 3 — Queue views

  • Three operational queues: Rejected, Transfer, Purchase
  • Filterable by status, assigned approver, branch, urgency
  • "Mark TO Created" / "Mark PO Created" buttons to capture EzyVet reference numbers
  • Full status progression: Approved → TO/PO Created → In Transfer / Ordered → Received → Closed

Week 4 — Notifications & edge cases

  • Status-change notifications to requesters at every transition
  • Approver reminders at 4h and 24h
  • Error handling: product not found, user not in mapping, sheet write failures
  • Admin slash command /inventory-admin for Michael's team to see open requests across all branches

Exit criteria: full request lifecycle works. Ready for pilot.

Phase 3 — Pilot + full rollout combined Jul 13–19 (1 week)

Days 1–2 — JLT pilot launch

  • 15-minute Loom + Slack canvas published in clinic-jlt-inventory
  • Category managers (Pauline, Ivan, Aneesh) onboarded the same day
  • Live monitoring, bug fixes in flight

Days 3–5 — Cascade rollout

  • If pilot is stable by day 2: roll out one clinic cluster per day (Modern Vet → Blue Oasis → Vienna Vet → TVRC → Mobile)
  • Same Loom + canvas reused across all branches — no fresh training material per clinic
  • Michael announces WhatsApp inventory groups sunset

Days 6–7 — Stabilisation & handover

  • Bug fixes from full-rollout feedback
  • Operational handover to Sinergi + Michael's team: hosting credentials, monitoring access, on-call procedure, env vars, deployment process, future-iteration guide

Exit criteria: live across the group. WhatsApp inventory requests stopped. Sinergi + Michael own ongoing operation.

Phase 4 — Post-handover enhancements After Jul 19 — Sinergi-owned

Candidates Sinergi may want to prioritise (not part of the MVP scope):

  • EzyVet API integration (auto-create PO/TO drafts from approved request lines)
  • Backup approver routing for leave coverage
  • Analytics dashboard: top-requested products, slowest approvers, request volume per branch
  • Slack workflow templates for repeat orders ("reorder last month's antibiotics")
  • Migration to a proper database if SKU count exceeds ~5k

9. Open questions we'd like Sinergi's input on

For Sinergi's team

  1. Backup approvers. MVP routes directly to a single category manager. If they're on leave, who covers? We propose deferring this to Phase 4 — does Sinergi agree, or would you like backup routing in MVP?
  2. Branch auto-fill on the request form. Auto-fill from user mapping, or require manual selection each time? We lean toward auto-fill with manual override.
  3. Multi-line requests. Spec implies one product per request line. Does Michael need a "request multiple products in one submission" flow, or strictly one-product-per-form?
  4. Receipt confirmation. Who confirms in Slack once a PO or TO is fulfilled and received? The receiver in the user mapping, or do you envision an EzyVet status webhook triggering it?
  5. Estimated price on request lines. Spec doesn't mention price. Including it would help category managers approve faster — do you want this in MVP?
  6. Reporting. What weekly/monthly reports does Michael's team want out of the repository? (Top requests, slowest approvers, branch volume, request-to-fulfilment time, etc.) Useful for shaping the Phase 4 analytics dashboard.
  7. EzyVet API. Do you know whether EzyVet exposes a usable API for PO/TO creation? Determines whether Phase 4 can automate the EzyVet handoff.
